# 剩余五部鲁迅代表作品统一接入设计

日期：2026-07-21  
项目：横眉·鲁迅文化数字展馆  
范围：《阿Q正传》《故乡》《祝福》《朝花夕拾》《野草》

## 1. 目标

在不重写详情页程序的前提下，沿用《狂人日记》样板，将其余五部代表作品统一接入现有作品详情体系。

本轮目标包括：

- 为五部作品补齐 1400～1800 个中文字符的原创导读与分析正文；
- 每部作品包含 2 处经原文核对的短引；
- 每部作品配置 5～6 条独立参考资料；
- 继续使用六张已完成的原创 WebP 封面；
- 保持收藏、引用跳转、返回正文和响应式布局不变；
- 由本地 JSON 自动生成 Supabase 批量迁移；
- 建立内容、迁移和整体回归测试。

## 3. 技术方案

采用“本地数据为唯一内容源，脚本生成数据库迁移”的方式。

数据流：

```text
assets/data/works.json
        │
        ├── 本地预览和数据库不可用时读取
        ├── 内容自动测试
        └── 批量迁移生成器
                    │
                    ▼
013_update_remaining_works_content.sql
                    │
                    ▼
Supabase public.contents
                    │
                    ▼
detail.js 安全解析并显示
```

不新增五个 HTML 页面，继续使用：

```text
detail.html?type=works&id=aq-zhengzhuan
detail.html?type=works&id=guxiang
detail.html?type=works&id=zhufu
detail.html?type=works&id=zhaohua-xishi
detail.html?type=works&id=yecao
```

继续复用：

- `detail.html`
- `assets/js/detail.js`
- `assets/js/detail-body-parser.js`
- `assets/css/detail.css`
- 收藏功能
- 引用跳转
- 返回正文
- 响应式布局

## 5. 正文格式与安全规则

继续使用安全轻量标记：

```text
## 二级标题

普通正文段落

> 经核对的短引。[1]

- 列表内容

正文资料编号：[2]
```

禁止正文出现：

- `<script>`
- `<iframe>`
- `onclick`
- 内联 `style`
- 任意原始 HTML

所有资料链接只保存于参考资料字段，由详情页通过安全 DOM API 创建链接。

## 6. 篇幅与引用要求

每部正文：

- 1400～1800 个中文字符；
- 恰好 2 个独立引文框；
- 4～6 个正文标题；
- 5～6 条独立参考资料；
- 引文后立即进行分析；
- 不使用大段原文代替原创正文；
- 不使用未经核实的网络摘抄；
- 正文资料编号必须能找到对应资料；
- 不保留未使用的孤立资料编号。

字符统计时排除：

- 轻量标记；
- 参考资料网址；
- 图片地址；
- 数据库元数据。

## 7. 参考资料策略

资料优先级：

1. 作品原始文本；
2. 鲁迅本人相关自述；
3. 正规文学机构、作家机构或公共数字文献；
4. 权威文学史和学术研究。

不采用：

- 无作者摘抄站；
- 营销号文章；
- 答案聚合页面；
- 无法确认版本的网络转录。

每条资料保存：

```json
{
  "id": 1,
  "title": "资料标题",
  "author": "作者或机构",
  "source": "网站或出版物",
  "url": "https://...",
  "accessed_at": "2026-07-21"
}
```

另建：

```text
docs/鲁迅代表作品内容资料与引用登记.md
```

每部作品登记：

- 作品名称；
- 两处短引主题；
- 短引原文出处；
- 背景与研究资料；
- 正文参考编号；
- 核对日期。

## 8. 批量迁移生成器

新增：

```text
scripts/generate-works-content-migration.cjs
```

只处理以下 slug：

```text
aq-zhengzhuan
guxiang
zhufu
zhaohua-xishi
yecao
```

职责：

1. 读取 `assets/data/works.json`；
2. 校验五个目标作品存在且唯一；
3. 生成 `013_update_remaining_works_content.sql`；
4. 安全处理单引号、换行、中文标点、JSON 元数据与网址；
5. 保证重复生成结果一致。

不会处理《狂人日记》和其他栏目。

## 9. 数据库迁移

迁移文件：

```text
supabase/migrations/013_update_remaining_works_content.sql
```

采用单一事务：

```sql
begin;

-- 更新五条作品记录
-- 校验实际更新数量为 5
-- 数量异常则抛出错误

commit;
```

允许更新：

- `summary`
- `body`
- 内容相关 `metadata`
- `updated_at`

明确不更新：

- `id`
- `slug`
- `content_type`
- `image_path`
- 图片授权元数据
- `sort_order`
- `published`
- `created_at`

缺少任意目标记录时，事务整体失败，避免部分更新。

`003_seed_initial_contents.sql` 同步重新生成，但不在当前数据库执行。当前 Supabase 只执行 `013`。

## 12. 异常处理与回滚

执行迁移前记录五条旧数据：

- slug；
- 正文长度；
- 参考资料数量；
- image_path；
- updated_at。

执行后再次查询同样字段。

迁移失败时：

- 依赖事务自动回滚；
- 不手工补执行部分 SQL；
- 修正数据或生成器后重新生成 `013`；
- 本地 `works.json` 保持为内容源。

单个页面异常时按顺序排查：

1. 本地 `works.json`；
2. Supabase 记录；
3. 浏览器缓存；
4. 内容解析结果。

不先修改公共详情页 CSS 或解析器，避免一部作品的问题影响其他栏目。
